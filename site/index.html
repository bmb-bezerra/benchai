Roteamento revisado

Default: Composer 2.5 Standard para implementação comum com plano claro.
Decisão cara: Claude Opus 4.8 para arquitetura, revisão qualitativa e contexto grande.
Terminal/debug: GPT-5.5 em esforço alto/xhigh para loop comando → log → correção.
Fable 5: manter como histórico/watchlist; não recomendar enquanto suspenso ou sem aprovação explícita.
Confidencial: contexto mínimo, Privacy Mode/ZDR confirmado, sem segredos, sem logs crus.

Diagnóstico da versão inicial

O material estava bem encaminhado, mas precisava sair de “relatório visual” para “ferramenta de decisão operacional”.

Correção crítica

O original recomendava Fable 5 para obras longas. Na versão nova, ele vira watchlist/quarentena por suspensão de acesso e exigência de governança.

Decisão editorial principal: evitei apresentar Fable 5 como recomendação ativa. Quando/ se o acesso voltar, ele deve ficar atrás de uma regra de aprovação, não como default.

Cenário Rota recomendada Modo Use quando Evite / escale

Perfis de modelos

Cards com status, melhor encaixe e pontos de atenção para uso por engenharia.

Governança por sensibilidade

A regra de modelo só é válida depois da regra de dados. Para uma página publicada no GitHub Pages, esse bloco evita interpretações perigosas.

Público / OSS

Baixo risco

Use a matriz normal. Ainda assim, não envie tokens, dumps, chaves ou dados pessoais.

Interno comum

Uso padrão

Privacy Mode ativo, contexto mínimo e validação por testes. Escolha modelos premium só se o custo de errar justificar.

Confidencial

Aprovação necessária

Sem logs crus, sem arquivos com segredo comercial, sem Fable/watchlist. Peça plano antes de editar.

Regulado / PII

Restrição forte

Não enviar dados reais. Trabalhe com exemplos sintéticos, trechos redigidos e autorização explícita.

Checklist antes de enviar contexto

✓
Remover segredos, tokens, chaves, dumps, arquivos .env e logs com PII.
✓
Confirmar Privacy Mode/ZDR e política do provedor antes de usar repositório de cliente.
✓
Pedir plano, arquivos afetados e riscos antes de permitir alterações multi-arquivo.
✓
Exigir validação: testes, lint, typecheck, comandos executados e resumo auditável.
✓
Verificar dependências sugeridas: pacote, mantenedor, licença, popularidade e typosquatting.

Riscos que faltavam ficar mais explícitos

Além de retenção, o playbook precisa tratar prompt injection indireto, divulgação de informação sensível, supply chain, output inseguro, excesso de agência e consumo sem limite.

Benchmarks: sinais, não oráculos

Os números sustentam a rota, mas não substituem teste no repositório da empresa. Compare apenas benchmarks equivalentes.

Leitura prática

O sinal mais consistente para terminal/debug favorece GPT-5.5. Composer 2.5 é forte no eixo eficiência/custo. Opus 4.8 é melhor tratado como modelo de julgamento e autonomia complexa. Fable 5 fica como histórico enquanto suspenso.

Fontes e rastreabilidade

As fontes ficam visíveis porque a página será usada por outras pessoas da empresa. Isso facilita revisão e evita “verdades” sem contexto.

Publicação no GitHub Pages

Este repositório já está preparado para publicar a pasta site/ como page de consulta interna.

Como publicar

1
Mantenha esta página em site/index.html.
2
Faça commit e push para a branch main.
3
Em Settings → Pages, selecione GitHub Actions como fonte.
4
Defina um owner para revisar fontes/modelos quinzenalmente ou quando houver release relevante.

Cuidados por ser page compartilhada

Não coloque nomes de clientes, URLs internas, arquitetura confidencial, tokens, exemplos reais de logs ou regras operacionais sensíveis no HTML. Mesmo se o repositório for privado, trate Pages como material de ampla circulação.
